百姓技术架构之我见

发表于 2019-06-09 | 分类于 总结 | 0 comments | 阅读次数
| 字数统计 2,533 | 阅读时长 9

写在前面的话

首先简单介绍一下百姓网。
能百度到的我不多说了。我提一些我印象里的关键词：

分类信息，程序员老板，58和赶集的手下败将，百度流量，佛系。

起心动念一段时间后，从百姓网离职了。奔赴新职之前，趁记忆犹新，记录一下我对百姓网技术架构的理解与感受。

以下内容写自2019年6月9日，已离职数日，加之个人理解可能有误，因此如与当前事实不符，请见谅。

既然是我见，那么首先介绍一下我在百姓网的十一个月时间里，是一个什么样的角色。
我的岗位是后端开发工程师，职级P1-1 -> P1-3，前后经历过数个项目，大致可以划分为如下几个阶段：

  1. 小程序的PHP后端
  2. Java+nodejs中台
  3. Python爬虫

其中第二个阶段，可以说是赶上了公司内部的整体架构迁移——从语言的角度来看，就是从php到 java，因此，我以主语言栈作为分界线来谈我眼中的百姓网的技术架构。

PHP时代

百姓网内部最大的代码仓库叫“镐京”，托管在GitHub，其对branch、pull request、CI/CD等的实践，我认为都是非常nice的。虽然我在这块儿的开发经验不多，但是仍然想着重聊聊。

镐京的前身是“朝歌”，看名字就知道，古得不得了，而且还是做“生意”的地方。——此处应有掌声。

对于百姓网业务形态的技术展现，我认为可以简单描述为“类目和地区下面的帖子”，当然了，更重要的是，它是一个网站。所以，镐京也是一个web应用，其所用的web框架是自研的，没记错的话是叫“alaya”，这一块儿没有研究不敢妄评，据说自研的原因是当时laravel还没出生（未考证）。不过对于开发者来说，web框架这一块需要关注的并不多，基本上只需要知道一个“router”是怎么用的即可。比较有意思的是镐京的“graph”层。

据说，在朝歌时期，即有数据库实体数据的“Node”层（非nodejs），而graph层，即是在Node之上，封装了数据节点之间的“relationship”。对开发人员来说，任何一条数据库中的数据（redis不计为数据库），不管是mysql还是mongodb，只需要知道它的全局id（prefix+id），只需要执行一个graph(gid)调用即可获取一个Node；如果它有children，那么只需要再load一次children就ok了。此外，全部的model定义，都通过工具生成，放到一个MData.php文件中。

镐京中已经尝试过服务化，而且用到了thrift。这一块儿我一直很迷惑（事实上我至今也没搞懂里面的道道），直到阅读了一篇文章《Thrift RPC在百姓网的实践之路》，才稍有所认知，且了解到前同事们其实是“魔改”了thrift协议的request结构来做“微服务链路追踪”的，而代码中的服务化是通过ServiceProxy来体现的。

从这篇文章里也可以看出，当时的百姓er们其实是非常注重开发体验的，镐京中衍生的许多工具也可以看出来这一点：

  • haojing命令行工具
  • 开发者独立的子域名站点（比如huangzhen.baixing.cn这个就是我可以用来debug的主站）
  • PhpStorm配置文件
  • 一个用flask写的自动安装composer依赖的工具
  • …

web应用里还有几个常见的东东，关联性不大，我铺大饼聊一下。

  • 定时任务
    草草地看过镐京里定时任务的实现，有一个每分钟运行一次的job，每次运行都会检查job文件夹下面的各个job（一个php类）是不是要执行。也就是镐京实现的定时任务的精度是1分钟。

  • 异步任务
    开发中没接触到，不了解。但是有几个redis实例是作为queue存在的。

  • 数据库
    仍然以MySQL为主，比如最关键的帖子是存MySQL的，似乎也没做sharding，单表过亿。
    MongoDB用得也很多，主要是各条业务线，这块的数据结构跟业务一样混乱，看了想打人。（据了解风控团队主要用Mongo，但是设计比较有持续性）
    前面聊到graph层，补充一点，graph层有一层封装，对上层屏蔽了数据库和ElasticSearch。对于热点数据的查询，几乎都是走ES的。

此外，镐京里的前端部分，我知道的是后端渲染blade模板（这一块在Java时代迁移至了eggjs + pug），更深入的得拜读一下贺老的文章才行了。guanli

个人认为，作为单体应用，镐京的整体架构还是不错的，可以做到“让没有经验的工程师写业务代码而无需考虑性能问题”，不足之处是在项目管理上没有能够做到很好地下线业务，导致老旧代码无人能懂无人敢动。

PHP时代镐京以外的项目，基本上都用上了成熟的Laravel框架，沿用php前后端不分离的套路，但从前端代码来看也引入了更多灵活的玩法。

Java时代

Java时代序幕已拉开，但仍然在路上。

可以明确的两个标志性事件是：

  1. 基于SpringCloud自研SpringTime
  2. 上k8s

镐京是托管在GitHub的，而后续逐渐转移到GitLab平台，并且基于GitHub hook那一套和jenkins的CI也基本上变成了GitLab自带的CI/CD体系：

  • CI跑项目打包，以及sonar代码检查，考察静态代码质量以及单元测试覆盖率，结果计入测试部门的考核成绩。
  • CD跑k8s部署命令。

Java时代的排头兵语言是Java8（现已支持Java11）；注册中心用的consul（CAP原则中满足CA）；分布式追踪用的Zipkin（这一块了解较少）；网关用的自研的gateway-service，Java写的同步阻塞网关服务，基本功能就是对url进行前缀路由匹配来实现服务路由；跨服务调用主要用Feign，即跳过网关走的是http协议；ORM主要用hibernate；数据库仍然主要用MySQL；监控用的prometheus，报警接入了企业微信、邮件、电话；监控数据可视化用的grafana；日志用的kibana；报错用的sentry；sql审核平台用的yearning（来自个人开发者）；redis审核平台用的cachecloud（来自搜狐）。

上k8s的不足是周边基础设施做得不太完善，开发体验比PHP时代应该说是复杂度上升、体验下降。但其带来的利好仍然是非常明显的。

值得一提的是，这一时期，整个测试团队兵强马壮起来，自动化测试似乎进展得不错。服务测试那边主要用的是Yapi，原去哪儿开源的API测试平台，除了可以直接测HTTP的API，还可以直接连接redis和mysql，但是支持的命令有限。

再说回Java时代的一个重要特征，nodejs登上舞台。我前后用过nuxt、express和eggjs这几个web框架，体验最好的是eggjs。事实上，前面也提到，前端团队事实上最后是采用了TS版的eggjs（还用上了graphql）。但是不得不说，eggjs并没有实现其node界的spring这一愿景。虽然我所在的team主要用的是vue+element-ui（而且都是用的ssr架构——基本上都是内部的后台管理系统），但是据了解公司内其他非前端team用到的轮子也还是五花八门，在此就不细表。

后来由于业务调整的原因，我脱离了Java开发，开始做Python爬虫，下面说说我所了解的python。

Python

前面说了我用python做爬虫，其实一开始我们尝试过用go，因此先简单聊一下go。
刚开始做爬虫（在爬虫技术组成立之前），抱着试试玩的心态，用go语言生态里的爬虫框架gocolly做了几个爬虫。框架本身功能比较弱，生态就更弱了，写爬虫的话严重不推荐。据悉内部有算法部门用go，但考虑到该部门人单力薄，因此可以说go是用得极少的。

再说回python，在写python爬虫之前，我是不知道公司内部有用python的……后来才得知，CRM团队是全系python的，而且还用的是flask+postgresql；此外应该还从文档上了解到某个team也用python，但是用的框架就很冷门了（makiki，完全没文档）。

与其他python team的同事交流，感觉Java时代里建设起来的那些基础设施尚未与python很好地融合（应该说这是人员组织架构导致的必然结果）。

在python爬虫的开发过程中，我了解到以下内容：

  • 包管理nexus不仅支持java，也支持python等，且支持删除已经上传的python包（pypi就不可以）；sonar同理亦然。注：内部java和python已经用上了nexus，但是目前只有java用上了sonar。
  • BI部门用redash（开源）以及神策（商业产品），数据打点走http协议的get请求（主站域名会有DNS问题导致丢点，get请求限制了单个数据的大小）。
  • 360出品的类redis存储系统pika不支持redis中的transaction（tidis支持），导致其在非常多的场景下无法替代redis。

最后聊聊代码以外的技术架构——在百姓网，都用哪些内部工具？（以下内容纯靠回忆）

  • Team周会：slashme
  • 敏捷开发项目管理：tapd
  • 邮件日历：微软office
  • 财务OA：xxx
  • 外呼系统：天润融通
  • 内推招聘：Mokka
  • 云服务：又拍、七牛、青云等
  • 工作沟通：企业微信
  • 文档：confluence，tapd => 语雀
